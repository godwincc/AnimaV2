## What This Project Is

A breeding roguelike deckbuilder ("Anima") inspired by Axie Infinity (breeding/parts) and Slay the Spire (roguelike run structure). Players breed creatures called **Anima** — framed as constructs/"Vessels," not literal animals — to obtain skill combinations, build a 3-Anima team, and run a node-based roguelike dungeon.

**Core design pillar:** reward SMART breeding, smart team composition, smart deckbuilding — not just raw stats.

**Current phase:** The entire backend loop is real and tested end-to-end (140/140 checks passing as of this session). Full art direction is established. All 6 core screens are fully locked (Hub, Weaving, Sanctum, Anima Profile, Delve, Collection). All 4 room-encounter screens are locked (Resource, Treasure, Shop). **The Combat screen is fully designed.** **The match result screen (Victory/Defeat/Retreat) and the new Anima Reveal screen are now also fully designed and backed by real implemented services this session** — see Match Result & Retreat System and Anima Reveal Screen below. Godot client implementation is the confirmed next phase; no more screen-design gates remain before it, though a Run/traversal-state model (see Known TODOs) is still needed to wire result screens to real data.

**Real pixel-art creature portraits are deferred** to a future outsourced pixel artist — Claude has no native raster/pixel image generation (architectural limitation of language models, not a subscription tier issue). Retrofitting real art into Godot later is confirmed easy: sprites are just texture references, so swapping a placeholder for a real PNG requires no structural changes.

## Tech Stack & Architecture

C#/.NET 10 backend, fully built and tested. Planned client: Godot (GDScript). Server: ASP.NET Core/SignalR, first-pass implemented this session — see Server / Accounts / Auth below.

**IMPORTANT:** there is NO real `CombatEventBus`. Every "event" is a direct inline check at a known checkpoint.

**Recurring gotcha:** root namespace "Anima" collides with the `Anima` class (CS0118). Prefer `using AnimaUnit = Anima.Core.Models.Anima;` as the default fix everywhere — more reliable than moving `using` above the namespace.

**Recurring gotcha:** `DelveRun.Team` and a Combat node's `CombatState.PlayerTeam` must be the **same `List<Anima>` reference**, never a rebuilt/cloned list — HP attrition across nodes works today only because `CurrentHp` mutations happen in place and propagate for free through shared object references. Any future Combat-node wiring (Godot client, Server project) that constructs `CombatState.PlayerTeam` from a copy of `DelveRun.Team` will silently break attrition. Same caution applies to `CurrentArtifacts`/`WispEarnedSoFar` on `DelveRun`, which are live passthroughs to `RunLedger.Artifacts`/`PersistentLedger` rather than separate copies — do not fork these into independent state that must be kept in sync by hand.

## Map Generation (LOCKED, VALIDATED — 500 seeded maps pass every rule)
7×15 grid + Boss. Fixed floors: 1=Combat, 9=Treasure, 15=Shop. **Random odds, Floors 6+ (REBALANCED, prior session): Combat 36%/Elite 16%/Resource 16%/Treasure 16%/Shop 16%/Reforge 0%.** Reforge stays a fully real, coded node type (targeting flow, `RollOffer`/`Accept`, tests) — this is a probability change only, not a removal; it simply won't currently generate on any map. **Confirmed safe:** a 0%-weight entry only works because Reforge is the last entry in the odds array (the weighted-roll loop always resolves to an earlier entry first, since `Random.NextDouble()` never returns 1.0) — this invariant is documented directly in `MapGenerator`'s code so a future reorder doesn't silently break it.

**Guaranteed Elite + Early-Game Elite Exclusion (LOCKED THIS SESSION, STS-inspired):**
1. **Floors 1-5: Elite excluded entirely.** Uses a separate `EarlyFloorTypeOdds` table (Combat 40%/Resource 20%/Treasure 20%/Shop 20%, Elite/Reforge omitted, not zero-weighted) instead of the Floor-6+ table. **Shop's 20% entry is intentionally left inert (LOCKED, confirmed deliberate — not a bug):** Shop is *also* independently banned on Floors 1-5 by the pre-existing `EliteShopMinFloorIndex` rule (a joint Elite+Shop floor gate from an earlier session, untouched by this one), so that 20% never actually manifests on a real map — the real, observable Floors-1-5 distribution is Combat/Resource/Treasure only, renormalized among just those three. Same precedent as Reforge's 0% entry in the main `TypeOdds` table: correct-on-paper dead weight is kept rather than removed, because it costs nothing and documents the "fair" redistribution intent even where it's practically moot.
2. **Floors 6-14: guaranteed ≥1 Elite.** Before the normal weighted roll runs anywhere, one random eligible node is force-assigned Elite directly (bypassing the odds table entirely). This is a floor, not a cap — the normal roll still includes Elite at 16% for every other node afterward, so a map can have more than 1 Elite, never fewer. **Flagged deviation from the literal brief:** eligibility is Floor 6-13, *not* Floor 6-14 as specified — Floor 14 had to be excluded because every Floor 14 node connects directly into Floor 15 (entirely fixed Shop), and Elite-Shop direct adjacency is already banned by the existing `NoAdjacentGroup` rule (the same rule that already makes Floor 14 unable to roll Elite under the normal weighted path today). Forcing Elite onto Floor 14 would have produced a guaranteed, unavoidable rule violation. **Sanity-checked against the Reforge-0%-safety invariant (asked for explicitly):** no interaction — the forced assignment is a direct `.Type =` write on one node, never touching `TypeOdds`, `allowed`, or the weighted-roll loop; it can only ever produce Elite, never Reforge, and doesn't change either odds array's order or contents.

Real re-validated data at the final odds (n=500 seeds, permanent batch validator built into the harness — 0 rule violations, 0 Elite occurrences on Floors 1-5, every map has ≥1 Elite, all reconfirmed not assumed): **avg/map — Combat 23.00, Elite 4.93, Resource 10.48, Treasure 14.37, Shop 7.70, Reforge 0.00, Boss 1.00.**

**Node types (6):** Combat/Elite/Resource(Wisp-only, deliberate)/Treasure/Shop/Reforge/Boss. Elite optional/skippable (still true from Floor 6 on — the guarantee is a minimum count, not a forced route through it).

## Reforge (LOCKED, VERIFIED — targeting flow + screen identity added this session)
Pay Wisp for ONE random part from ANY color. Accept(40/80Wisp)=swap+lose Augment; Decline=free. Run-only (swap lasts only for the current Delve).

**Targeting flow (NEW THIS SESSION — mechanically essential, not optional, since Accept previously had no defined target to apply the swap to):**
1. Enter Reforge, one random part is rolled (random color + random skill within that color) and revealed.
2. Player picks **which of their 3 Anima**, then **which of that Anima's slots** (Head/Frame/Tail/Crest — Crest included, even though it's excluded from the deck, since Reforge swaps the actual Thread and Crest still has a real passive Thread to overwrite) to socket it into.
3. Accept commits the swap into that slot for the rest of this Delve only; Decline is free, nothing changes.
**Open question, not yet resolved:** what differentiates the 40 vs 80 Wisp cost tiers — needs a decision before this is fully spec-complete.

## Weaving System (LOCKED, VERIFIED — real Axie Infinity mechanics as direct inspiration)
Per part slot (ALL FOUR: Head/Frame/Tail/Crest, no exceptions): pool 6 candidate genes (D/R1/R2 × 2 parents), weighted 37.5%/9.375%/3.125% → winner becomes offspring's new Dominant. Two more rolls fill R1/R2. Mutation: 10%, R1/R2 only. Color: flat 50/50, 100% if same-color parents. Hybrid trigger: both parents pure + correct pairing = 33%.
Sibling restriction (copies Axie): only parent-offspring + full-sibling blocked. Weave Count: max 5, cost curve 50/100/175/275/400. Echo: twin-Vessel, 5% spontaneous or guaranteed via 5 Echo Shards.
**Naming:** players name their own Animas — mandatory prompt on Weave creation. Starter trio (Ember/Boulder/Sprout) keeps hardcoded names, no prompt.

**R1/R2 discard BUGFIX (found + fixed this session, predates this session — not newly introduced):** `AnimaMaterializationService.Build` (both the Weave and Boss-hatch overloads) only ever carried each part's Dominant skill into the materialized `Models.Anima`, silently dropping R1/R2 the instant a Vessel was created. This wasn't just a Profile-display gap — a materialized Anima that later became a parent had no R1/R2 of its own for `WeavingService.ResolvePart`'s 6-gene weighted pool to draw from, so **every 2nd-generation Weave was silently working off a broken (Dominant-only) pool** until this fix. Fixed by: (1) adding nullable `HeadR1`/`HeadR2`/`FrameR1`/`FrameR2`/`TailR1`/`TailR2`/`CrestR1`/`CrestR2` fields to `Models.Anima` (nullable, not `required` — the starter trio never goes through `Build` and has no recorded hidden genes at all, unlike Weave/Boss-hatch Anima); (2) `Build` now populates all three genes per part instead of just Dominant; (3) new `GenomeFactory.CreateGenome(anima)` is the single entry point for "get this roster Anima's `AnimaGenome` so it can be a Weave parent" — real stored R1/R2 (`ExtractGenome`) for anything materialized via `Build`, falling back to the pre-existing `CreateFounderGenome` synthesized placeholder only for the starter trio (detected via `HeadR1 is null`). No EF Core migration was needed for any of this: `PersistedAnimaEntity` stores the whole `Anima` as an opaque `AnimaJson` blob (same as the pre-existing `WeaveCount`/`ParentAId` fields), so the new nullable fields flow through automatically — verified for real with a live round-trip against the dev Postgres container, not just a design-time check.

**Echo Twin link (NEW this session):** `Models.Anima.EchoTwinId` (nullable string) is now set on both sides of an Echo-triggered Weave's Primary/Twin pair (by `GameHub.ConfirmWeave`, after both are materialized and their real Ids exist). This closes a real gap: `WeavingService.AttemptWeave`'s Twin is bred from the exact same parent pair as its Primary in the same call, so before this fix a genuine Echo Twin was indistinguishable from an ordinary full sibling via `ParentAId`/`ParentBId` alone (both would just look like two full siblings — correctly so, since an Echo Twin genuinely IS a full sibling too, just additionally linked). `WeavingService.AreFullSiblings`/`IsDirectParentChild` are untouched and still validate correctly against the new fields (they only ever read `ParentAId`/`ParentBId`, which `EchoTwinId`/R1/R2 don't affect) — confirmed via the existing 140-check console harness suite, still 140/140 after this change.

**Weaving is now ported onto `GameHub`** (see Server / Accounts / Auth below) — `AttemptWeave`/`ConfirmWeave`, both real economic effects (Wisp/Echo Shard spend, WeaveCount increment) committing immediately in `AttemptWeave`, with only the resulting Vessel(s)' roster row waiting on `ConfirmWeave`'s mandatory naming step (matching the Anima Reveal screen's "name is not skippable" contract).

**Pending-Weave persistence gap (found + fixed, Phase 3 audit):** the guard that blocks a second `AttemptWeave` while one is unnamed used to live only on `PlayerSession` (in-memory, keyed off a SignalR `ConnectionId`) — a dropped connection between `AttemptWeave` and `ConfirmWeave` dropped the whole `PlayerSession` too, silently resetting the guard AND permanently losing the already-paid-for roll (Wisp/Echo Shard spent, a WeaveCount charge consumed, no Vessel to show for it — WeaveCount is capped at 5 uses per parent, so this was a real, non-trivial loss). Fixed with a new `PendingWeaveEntity`/`PendingWeaveRepository` (EF Core migration `AddPendingWeave`, applied and round-trip-verified against the local Postgres dev container, including a real `WeavingResult` genome) — `AttemptWeave` now persists the pending Weave immediately, `PlayerSessionRegistry.CreateAsync` reloads it on every (re)connect, and a new `GetPendingWeave` method lets a reconnected client re-render the Anima Reveal screen before confirming. `ConfirmWeave` deletes the row on success. Deliberately NOT extended to `DelveRun` or pending Ember pickups in the same pass — see their own sections for why those stayed in-memory-only (lower stakes, not the thing that was actually broken).

## Ember & Augment System (REWORKED THIS SESSION — replaces prior persistent-bank model)
Ember is no longer a stored/banked resource. It is a momentary "chance to augment," resolved individually at pickup or purchase.

**Drops:**
- Combat: 1 Ember, random color, guaranteed.
- Elite: 1 Ember guaranteed + independent 25% chance each for a 2nd and 3rd Ember (max 3).
- Resource: 15% chance of 1 bonus Ember, additive to the base 30 Wisp.

**Pickup flow (per Ember, sequential if multiple dropped — never batched):**
- **Augment now** — opens the Augment page scoped to that Ember's color. Player picks a skill whose own archetype color matches the Ember's color (see Augment Eligibility below — this is skill color, not Anima body color). Consumes the Ember, charges the Wisp tier for that skill's next augment slot.
- **Convert to Wisp** — flat 15 Wisp, Ember discarded, no augment.

**Augment cost** (replaces the old 2/4/7 Ember curve entirely): 1 Ember (consumed) + a Wisp tier based on how many augments that specific skill already has — 1st augment: 15 Wisp, 2nd: 30 Wisp, 3rd: 50 Wisp. Max 3 augments/skill unchanged. The 4 augment types (Increase Effect +20%, AoE Damage 50%, Decrease Cost -1, Extend +1 Charges) unchanged.

**Augment eligibility (REWORKED):** checked against the skill's OWN archetype color (Crimson: Ember/Reaper/Marksman, Onyx: Boulder/Aegis/Warden, Verdant: Sprout/Thicket/Lotus, Azure: Shade/Anchor/Veil), NOT the Anima's body color. This means a single Anima's 4 skills (Head/Frame/Tail/Crest) can each require a different Ember color if bred from mixed-color parts (hybrids Vulcan/Mirage, or any atypical part lineage). This is the correct general rule, not a hybrid-specific patch — it retires an earlier "either parent color works for hybrids" proposal that's no longer needed.

**Pickup flow ported onto `GameHub` (Resource's own drop only, this session):** `PlayerSession.PendingEmbers` is a real FIFO queue (not a single slot), specifically because the locked "sequential if multiple dropped — never batched" rule above means Elite's up-to-3 drops (Phase 5) need the same mechanism. `ConvertPendingEmberToWisp`/`AugmentPendingEmber` resolve the front of the queue only. **Deliberately in-memory only, NOT given the PendingWeave DB-backed fix** (see Weaving System above) — an unresolved Ember has no stored value anywhere by design (this section's own opening line), so losing one to a dropped connection costs at most one Augment or 15 Wisp, far below a paid-for Weave's stakes. A conscious proportionality call, not an oversight.

### Turn Resolution (LOCKED THIS SESSION — Darkest Dungeon-style, not STS batch-commit)
Turn order for the Round is rolled once at Round start (all 6 combatants), shown as a queue. Combatants act **one at a time in that rolled order**, interleaving player and enemy turns — NOT a "queue all your actions, then watch the round resolve" STS-style batch. This was a deliberate choice: the skill kit has heavy reactive/interrupt design (Intercept, Guard Strike, Deflect, Ward, Retribution, Vengeance, Last Laugh) that only makes sense if the player can see what already happened this Round before acting. Enemy intent telegraphing (STS-style "show what the enemy will do before they do it") was explicitly REMOVED given one-by-one resolution — the player sees the real action play out live instead.

Player can **Pass** a given Anima's turn (opts out of playing anything that turn) via an explicit Pass action.

## Combat Screen Design (NEW THIS SESSION — fully designed, not yet implemented in Godot)

**Layout, top to bottom:**
1. HUD row: Round counter (also shows node type, e.g. "Round 3 — Combat"), Wisp count (sparkle icon, never a coin), Artifacts row (icons, hover for description — see tooltip pattern below).
2. Divider line.
3. **Arena**: darker inset panel (`rgba(0,0,0,0.42)` over the base gradient) clearly separated from the rest of the HUD — this was an explicit request to make the arena "pop" visually. Player team left, enemy team right, no "VS" divider text. Each side's 3 Anima/enemies arranged front-to-back with front nearest the center gap (position 1=front closest to center, 2=mid, 3=back, matching the existing position table).
4. Each combatant card: portrait, name, HP bar (color-coded: green >50%, amber/gold 25–50%, red ≤25%), HP numbers, a buff/debuff icon row (FIXED HEIGHT reserved always, even when empty, so...), a **position-number line (1/2/3) that sits at the same vertical height for every combatant regardless of buff count** — this required reserving fixed height for the buff row rather than letting it collapse when empty. Dead/removed combatants show as an empty slot at the same total height (position held, portrait gone), with a darkened line and dim position number — no dashed placeholder box.
5. The currently-active combatant's position line is highlighted gold, tied to the turn order queue's highlighted entry.
6. Divider, then a **message area** (NOT a floating tooltip, NOT a top banner) — a single docked line of text used for THREE things: (a) buff/debuff and Artifact hover descriptions, (b) the "choose a target" instruction when a targeted card is selected, (c) the Confirm/Cancel prompt before a card resolves. This consolidation was deliberate — floating tooltips/popups positioned near edge-of-screen elements (e.g. the rightmost enemy, or the Artifacts row in the top-right corner) kept overflowing the viewport with center- or edge-anchored CSS; a single area that's part of normal document flow cannot overflow by construction. Hovering something temporarily overrides the message area's content and reverts to whatever the "current state" message was (targeting instruction, confirm prompt, or blank) on mouse-leave.
7. Divider, then: turn order queue (LEFT column, vertical list of full names — not initials — showing the FULL Round's rolled order, gold left-border-accent for player-side entries, coal-gray (`#8a8175`, matching enemy portrait color) for enemy-side, current/next actor highlighted) alongside Energy pips + Deck/Discard counts + Pass button (all in one row) + the hand of cards.
8. Hand cards: vertical card layout, cost pip (circle, color-coded to the skill's archetype color), placeholder art block, skill name, effect text. Unaffordable cards (insufficient Energy) are dimmed with a red X overlay and not clickable. Hand max is 5 (6 with Weaver's Thread) — deck/discard counts sit inline with Energy (not as a side column next to the hand) specifically to leave room for a 6th card without crowding.
9. Combat log: bottom of screen, scrollable (deliberate exception to the general "no nested scrolling" UI convention used elsewhere — a combat log needs full history, not just the last few lines), logs ALL combat events including AI/enemy turns and system events (Round starts, Enrage escalating), color-coded by event type (damage/heal/shield/neutral).

**Targeting/confirm flow:**
- Clicking a hand card: if self-targeting, goes straight to the Confirm prompt (in the message area) using the acting Anima as the implicit target. If it needs an enemy/ally target, outlines ONLY the legal target set on the arena (not all combatants) and shows a "choose a target" instruction in the message area.
- Clicking a legal target opens the Confirm prompt in the message area: "Play [skill] on [target]? Confirm / Cancel." Nothing resolves until Confirm is pressed. Cancel fully backs out of targeting mode. This confirm step was explicitly requested as a misclick-prevention measure.
- On Confirm: a floating +/- number rises and fades above the target's portrait, plus a brief colored flash on the portrait (red=damage, green=heal, silver-blue=shield). IMPORTANT implementation note: in the real client this feedback must fire only after the server/combat engine confirms the action's actual resolved result (including anything a reactive skill like Deflect/Vengeance changes), not optimistically on click — the mockup fires it optimistically only because there's no real backend behind it.

**Enrage display:** shown as a buff/debuff icon (flame) in the affected combatant's own status row, NOT a separate standalone HUD badge — this matters because Enrage is per-combatant (typically Boss/Elite), not a screen-wide state.

**Hover tooltips (buffs/debuffs, Artifacts):** originally floating positioned popups, retired in favor of the docked message area (see point 6 above) specifically because floating tooltips centered on edge-of-screen elements overflowed the viewport and a per-element edge-detection fix (right-align near the right edge, left-align near the left edge) proved fragile in testing. The docked message area is the intended real pattern — worth reusing anywhere else in the UI that has this same edge-tooltip-overflow risk (Collection's Artifact list, Sanctum cards, etc.).

**Formerly deferred here, now resolved (stale cross-reference cleaned up this session):** the win/loss transition and Elite Ember-pickup placement questions that used to be listed as "not yet designed" were both resolved by the Match Result & Retreat System section below — see there instead.

## Boss Reward & Anima Reveal Screen (NEW THIS SESSION — fully designed AND implemented)

**Boss reward, final structure:** guaranteed Wisp (300) + guaranteed hatched Anima (resolves via Anima Reveal screen, not a banked resource — the old `ResourceType.Vessel` concept is fully retired, removed from the codebase entirely) + guaranteed Echo Shard.

**Boss-hatch Anima roll logic (no lineage/parents involved), implemented in `BossHatchService.Roll`:**
1. Roll body color, flat 25% each, Crimson/Onyx/Verdant/Azure. **No hybrids** — Vulcan/Mirage excluded entirely.
2. Roll each of the 4 parts' color independently, weighted **55% toward the Step 1 body color, 15% each for the other 3.**
3. Roll each part's Dominant, R1, and R2 as three independent 55/15/15/15-weighted rolls (via `SkillPool.RollRandom(Part, AnimaColor, Random)`), then pick the actual skill from that rolled color's pool. This makes Boss-hatch genomes **structurally identical to Weave-produced ones** — `IsFullyPure`, `WeavingService`, and materialization all work with zero special-casing (tested: `IsFullyPure` lands at the expected 0.55⁴ ≈ 9.15%).
4. Gen is fixed at **Gen 1** (wild origin point, same tier as starter trio conceptually, but goes through the normal mandatory naming prompt — starter trio remains the sole hardcoded-name exception).

Confirm commits the name, adds the Anima to the Sanctum roster, and returns to the triggering flow (Boss Victory → Delve Complete; Weaving → normal screen state).

- `Models.Anima` gains required `Name` (mutable, distinct from Id — `DisplayName` now returns `Name`) and `Gen` (int) fields. All 13 `SampleAnimas` factories updated (Name = Id, Gen = 1).
- `AnimaMaterializationService` (new, shared by both triggers): Weave overload computes `Gen = max(parentA.Gen, parentB.Gen) + 1` and records lineage; Boss-hatch overload fixes `Gen = 1`, no parents, looks up Stats via `WeavingService.ColorStats`. ## DelveRun — Run/Traversal State Model (NEW THIS SESSION — implemented)

`Anima.Core.Run.DelveRun` is the missing glue between node visits — the "save file for a single in-progress run." Scope was deliberately kept minimal (in-memory only, no save/resume, no mid-combat state):

- **`CurrentNode`/`AvailableNodes`/`TryMoveTo`** — map position on the 7×15+Boss grid. Boss surfaces automatically as an available node with no special-casing needed.
- **`ClearedNodes`** — caller-driven (not automatic on move), so the map can render visited vs. available.
- **`Team`** — a plain `List<Anima>`, the same list a Combat node's `CombatState.PlayerTeam` is built from. HP attrition needed zero separate tracking as a result — `CurrentHp` mutations persist for free through the shared reference, **as long as nothing ever rebuilds/clones this list** (see gotcha above).
- **`CurrentArtifacts`/`WispEarnedSoFar`** — live passthroughs to `RunLedger.Artifacts`/`PersistentLedger`, not separate copies, to avoid forking state that has to be hand-synced.
- **`DelveRun.Start(...)`** — the one real call site that sets `RunLedger.WispAtDelveStart` from the current `PersistentLedger` balance. **Bugfix note:** `WispAtDelveStart` existed since the Match Result & Retreat session but nothing outside test code ever actually set it — every real Delve would have diffed Wisp-earned against a stale 0 before this fix.

**Explicitly out of scope for this pass (by design, not oversight):** no mid-combat state (an active Combat node owns its own state; `DelveRun` just receives the result on resolution), no save/resume across app restarts (matches `SanctumRoster`/`PersistentLedger`'s existing no-save/load caveat), no multiplayer/concurrency handling (single active `DelveRun` at a time).

## Combat System — Server Port (Phase 5a, NEW THIS SESSION — core loop only, no rewards)

Ports the core Combat loop onto `GameHub`. **Deliberately stops the moment a winner is determined** — Victory/Defeat rewards (Wisp/Ember/Vessel/Echo Shard), `DelveEndService`, Boss-hatch + Anima Reveal, and the Artifact win-count stat are all Phase 5b, not touched here.

**Audit findings (confirmed by reading `CombatEngine`/`CombatState` and the console harness's own `RunDelveFight`, not assumed):**
1. **`RunRound()` is one synchronous, uninterruptible loop** — `RoundStartPhase → InitiativePhase → ActionPhase (loops every combatant, both sides, interleaved) → RoundEndPhase`, all in a single call. Player decisions came from a synchronous `Func<Anima, CombatState, Skill?>` callback (`ChoosePlayerSkill`) invoked mid-loop — fine for the console harness's scripted heuristic AI (`ChooseDelveSkill`), architecturally incompatible with a real human player needing an actual network round-trip per decision. This is genuine restructuring work, not a thin wrapper — confirmed, not assumed, before writing any hub code.
2. **No real player target choice existed anywhere in Core.** `CombatEngine.SelectTarget` is a fully deterministic static method (Marked-priority → front position, or lowest-HP, or first-alive). `TargetType.ChosenEnemy`/`ChosenAny` exist as enum values, and `SampleAnimas.cs`'s own comments literally describe them as "stands in for a real player choice" — but `SelectTarget`'s switch never distinguished them from the fully-automatic types; they fell through to the same `_ => alive.FirstOrDefault()` default. `ChoosePlayerSkill`'s signature only ever returns a `Skill`, never a target — the console harness's `ChooseDelveSkill` never picks one either. A second, independent architectural gap from turn-pausing.
3. Initiative order (`_currentInitiativeOrder`, needed by Ambush's "acted last" check) lived on a **private `CombatEngine` instance field**, not on `CombatState` — would not survive a fresh `CombatEngine` being reconstructed per hub call (the pattern every other `GameHub` method already uses). Moved onto `CombatState.TurnOrder`.
4. Win/loss was checked by the **caller**, between external `RunRound()` calls (`RoundEndPhase`'s own comment: "Win/loss condition is checked by the caller after RunRound() returns"; confirmed via the harness's own `while (... PlayerTeam.Any... && EnemyTeam.Any...)` loop) — never inside the engine, so it couldn't stop mid-Round.
5. Confirmed real and correct as documented, no changes needed to their *rules*, only to how they're *driven*: the 27-card deck build (Head/Frame/Tail ×3, Crest excluded), Energy +3/cap 9, Vanguard's Bell's Round-1 bonus (already fires once via `OnCombatStart`, no change needed), Enrage escalation, Sifting Stone/Barrier Stone, hand top-up-not-reshuffle.

**Resumable engine additions (`Anima.Core.Combat`), all additive — `RunRound()`/`ChoosePlayerSkill`/algorithmic-only targeting are UNTOUCHED in behavior, confirmed by the console harness's 140/140 still passing after every change:**
- `CombatState` gains `TurnOrder`/`TurnIndex`/`CurrentActor` (computed). `CombatOutcome` enum (`InProgress`/`Victory`/`Defeat`), computed by `CombatEngine.GetOutcome(state)`.
- `ExecutePlayerSkill` is `ResolvePlayerTurn`'s old body, factored out and given an `explicitTarget` parameter — `ResolvePlayerTurn` (RunRound's path) still always passes `null`, reproducing the original algorithmic-only behavior byte-for-byte. `explicitTarget` is threaded down through `ResolveSkill`/`ResolveAttack`/`ResolveMove`/`ResolveBuff`/`ResolveDebuff`/`ResolveHeal` as `explicitTarget ?? SelectTarget(...)` at each call site — when set, it overrides the algorithmic pick; when null (every existing call site), identical to before.
- `CombatEngine.AdvanceUntilPlayerActionNeeded()` — the actual resumability primitive. Auto-resolves every Enemy turn and Round transition (top-up draw, Energy refill, Enrage checks, a fresh Initiative roll — reusing `RoundStartPhase`/`RoundEndPhase`/`InitiativePhase` directly, not duplicated) until either a living player Anima is next to act (pauses, returns `InProgress`) or the fight ends (`Victory`/`Defeat`). Called once right after `StartCombat()` (resolves anything before Round 1's first player turn, e.g. a faster enemy going first) and again after every player action.
- `CombatEngine.ResolvePlayerAction(anima, skill, explicitTarget)`/`ResolvePlayerPass(anima)` — the new validated entry points `GameHub.SubmitAction` calls. Unlike `ExecutePlayerSkill`'s original lenient behavior (silently treats an unusable skill as a pass, never checks Energy affordability or deck ownership — always true in practice only because `ChoosePlayerSkill`'s caller was trusted to pre-filter), these throw `InvalidOperationException` on: wrong turn, skill not owned by the acting Anima, not usable from current position, insufficient Energy. **Deliberate, necessary difference**: the "caller" of a hub method is an untrusted network client, not a scripted harness, so these can no longer be assumptions.
- `CombatEngine.GetLegalTargets(actor, skill)` — computes the real legal-target set per `TargetType`, for the client's highlight-then-confirm flow (Combat Screen Design). Most `TargetType`s are actually algorithmic/automatic (returns the one resolved target, not an empty set, so "outline the legal target set, click to confirm" still works uniformly even for a single-option skill); `ChosenEnemy`/`ChosenAny` are the only two genuine "player picks among several" types (confirmed via finding #2 above) and return the full living opposing/friendly team respectively.

**`GameHub` additions:** `StartCombat`/`SubmitAction`/`GetLegalTargets`/`GetCombatState`, all in `CombatStatus`'s shared response shape (mirrors the one-shared-DTO pattern `DelveEndResult`/`ShopStockSnapshot` already use). Request-response only, matching every other `GameHub` method — no SignalR server-push in this pass; real push-based live turn streaming is a decision for whenever the actual Godot client build reveals whether it's needed, not built speculatively now. `SubmitAction` validates whose turn it is + the target's legality, resolves the one action, then auto-resolves consecutive enemy turns via `AdvanceUntilPlayerActionNeeded`, returning the full resulting state + event log in one response. `BuildEncounter` mirrors the console harness's own Delve-simulation convention exactly (Combat = Grovehide+Quillfang at positions 1/2, Boss = Warden of the Hollow) with one deliberate simplification: Elite is a random 50/50 Sentinel/LeechMother pick instead of the harness's alternating counter (that counter exists only for even test coverage, not a real gameplay rule). `PlayerSession.ActiveCombat` (`CombatState?`) is **in-memory only**, same lifetime as `ActiveDelveRun` — a deliberate choice, NOT the `PendingWeave`/`PendingPurchasedEmber` DB-backed treatment: `ActiveDelveRun` (Combat's own enclosing container) is *already* in-memory-only and lost on disconnect today, so persisting `CombatState` while its enclosing `DelveRun` doesn't would be a strange half-measure (a resumed fight pointing at a Delve that no longer exists); HP loss taken mid-fight is already durable regardless (same Anima instances `SanctumRosterRepository` saves after every action — see the reference-identity point below), and Phase 5a grants no rewards/spends no currency entering or resolving a fight, unlike a paid-for Weave or a bought Ember.

**Phase 5a closeout audit (this session, follow-up to the Phase 5a commit above):**
1. **Target-legality validation was audited and confirmed already real, not missing.** The original 5a scope called for server-side rejection of an illegal `explicitTarget` as misclick/anti-cheat protection. Read closely, not assumed: `GameHub.SubmitAction` (not `CombatEngine.ResolvePlayerAction` itself, which trusts its caller) computes `GetLegalTargets(anima, skill)` fresh from server-side `CombatState` on every call and throws `HubException("Illegal target for that skill.")` if the client's target isn't in that set (or is missing when one is required) — before `ResolvePlayerAction` ever runs. For `AllEnemies`/`AllAllies` skills the check is skipped (the legal set is empty by design), which is safe rather than a hole: `ResolveAttack`/`ResolveHeal`/`ResolveBuff`'s AoE branches never read `explicitTarget` at all, so a client sending a bogus target for one of those has no effect either way. No code change was needed here.
2. **Silent Chime's engine-instance-field bug (flagged as a known gap in the Phase 5a commit above) is now fixed.** `_pendingExtraActionAnima` moved off `CombatEngine` onto `CombatState.PendingSilentChimeTarget`, the same treatment `TurnOrder` already got. `TryActivateSilentChime` now writes to `_state.PendingSilentChimeTarget`; the legacy `ActionPhase` loop (RunRound's synchronous path) reads it from the same place, unchanged in behavior — confirmed via the existing console-harness Silent Chime checks, still passing after the change. The NEW resumable path (`ResolvePlayerAction`/`ResolvePlayerPass`) now shares a private `AdvanceTurnOrConsumeSilentChime(anima)` helper: if `anima` is the pending target, it's cleared and `TurnIndex` is deliberately NOT advanced (so `CurrentActor` stays on the same Anima for the granted extra action); otherwise `TurnIndex` advances normally. **Verified via a throwaway in-process harness (`Anima.SilentChimeVerify`, deleted after use, same spirit as Phase 5a's own `Anima.Phase5aHarness`)**, specifically using a FRESH `CombatEngine` instance for every step (activation, first action, extra action, re-activation attempt) to prove the fix actually survives what the bug was about — a real hub call reconstructing the engine between steps: activation on engine #2 was correctly visible to a completely separate engine #3's `ResolvePlayerAction` call; `CurrentActor` stayed on the targeted Anima across that call (extra action granted); a second `ResolvePlayerAction` on yet another fresh engine #4 consumed the extra action and advanced `TurnIndex` normally; a final fresh engine #5's re-activation attempt correctly failed (single-use per Delve, unaffected by this fix). Console harness reconfirmed at 140/140 after the change. Silent Chime is still not wired to any `GameHub` method (that part of the original gap remains — this fix only removes the structural blocker so wiring it up won't silently lose the activation).

**Verified via a throwaway in-process harness (`Anima.Phase5aHarness`, deleted after use)** driving `GameHub`'s real Combat methods directly against the real dev Postgres container (`Hub.Context`'s public setter stood in for a real SignalR transport, same pattern Microsoft's own SignalR unit-testing guidance uses) — no new EF migration needed, since `ActiveCombat` is intentionally in-memory-only:
- **`DelveRun.Team`/`CombatState.PlayerTeam` reference identity** (the exact landmine flagged in this file's own gotcha, and the same class of gap the pending-Weave/Ember-purchase audits kept finding elsewhere) — confirmed by `ReferenceEquals`, not assumed: same `List<Anima>` reference AND every individual `Anima` instance reference-identical. `GameHub.StartCombat` builds `CombatState.PlayerTeam = run.Team` directly.
- **Full combat to Victory** (real play via `GetLegalTargets`+`SubmitAction`): won in 5 Rounds against a basic Combat encounter.
- **Full combat to Defeat** (team submits Pass every turn): wiped in 79 Rounds — confirms the engine doesn't stall or hang even under a maximally-passive player, and that `AdvanceUntilPlayerActionNeeded`'s Round-transition branch is stable across a long run.
- **Long fight, Round-boundary + Enrage check**: walked a real generated `DelveRun` map straight to the Boss node (`MoveToNode` doesn't require the current node to be cleared first — confirmed via code reading, used deliberately here to skip straight past 14 floors' worth of intervening nodes) and fought Warden of the Hollow for real. 26 distinct Round numbers observed across the fight (confirms repeated Round-transitions genuinely fired through the new resumable path, not just the fight ending), and the real event log captured `"Warden of the Hollow becomes ENRAGED!"` — Enrage escalation confirmed firing correctly through `AdvanceUntilPlayerActionNeeded`, not just through the old `RunRound` path.

**Known gaps flagged by this phase:**
1. ~~Silent Chime (`CombatEngine.TryActivateSilentChime`) is not wired to any hub method~~ — the underlying structural blocker (the engine-instance-field bug) is RESOLVED, see Phase 5a closeout audit above. **Still not actually wired to a `GameHub` method** — that part of the gap remains, since it wasn't in this phase's (or the closeout's) requested method list; it's a pre-Round activation hook, not itself part of the per-actor turn loop.
2. **`GetLegalTargets` doesn't check card ownership** (whether the requested hand card belongs to the current actor) — deliberately lenient, since it's read-only/nothing committed; `SubmitAction` is the actual enforcement point. A client previewing targets for a card it can't actually play is harmless.
3. Reforge stays deliberately deferred (0% map odds, unresolved open items — unchanged by this phase).

## Combat Rewards, Boss-hatch, and Match Result — Server Port (Phase 5b, NEW THIS SESSION)

**Audit findings (confirmed by reading `DelveEndService`/`BossHatchService`/`AnimaMaterializationService`/`RewardService`/`ArtifactService`, not assumed):**
1. **`DelveEndService.ResolveDefeat`/`ResolveRetreat` were already real and correct**, exactly as CLAUDE.md described — no Core changes needed. Both diff `PersistentLedger`'s current balance against `RunLedger.WispAtDelveStart`, not total balance, so the "keep 50%/100% of THIS run's Wisp" design already worked correctly at the Core layer.
2. **`RunLedger.WispAtDelveStart` read-safety, specifically asked about:** confirmed safe. `PlayerSession.ActiveDelveRun` (holding `RunLedger`) is a single in-memory object reference held for the life of one connection — every `GameHub` method call on that connection reads the SAME `RunLedger` instance (no per-call reconstruction the way `CombatState`'s owning `CombatEngine` gets reconstructed each `SubmitAction` call), so there's no "hub-call lifecycle" risk of the snapshot going stale between Delve-start and end-of-run math. The only real risk is the **already-known, pre-existing "`ActiveDelveRun` is entirely in-memory and lost on any disconnect"** limitation (same class of risk `ActiveCombat` already carries, documented since Phase 5a) — not new to this phase, and out of scope to fix here (would mean DB-backing the whole `DelveRun`, a materially larger change than what was asked).
3. **`BossHatchService.Roll`/`AnimaMaterializationService`'s Boss-hatch overload were already real and correct** — genuinely structurally identical to a Weave-produced genome (Dominant+R1+R2 per part), needing zero special-casing to plug into the same `ToPreview`/reveal-screen pipeline Weaving already uses.
4. **`RewardService.GrantCombatWin`/`GrantEliteWin`/`GrantBossWin` were already real and correct**, including the exact reward tiers (Combat 50 Wisp+1 Ember, Elite 120 Wisp+1-3 Ember+25% Vessel Shard, Boss 300 Wisp+guaranteed Echo Shard, **no Ember from Boss** — confirmed by reading `GrantBossWin` itself, not assumed from the design doc). The hatched Anima itself is deliberately NOT resolved by `RewardService` (by design, per its own comment) — the hub is what glues `GrantBossWin` + `BossHatchService.Roll` together.
5. **The Artifact win-count write path (`AccountArtifactStatEntity.DelvesWonWithCount`) had no write method at all** — `AccountArtifactStatRepository` only had `RecordDiscoveryAsync`. Added `RecordWinAsync` (upserts, though in practice a row should already exist from discovery).

**New DB-backed pending state (`PendingBossHatch`), mirroring `PendingWeave` exactly:** a Boss Victory's guaranteed hatched Anima is a real, already-granted-in-substance reward (the fight is won, Wisp/Echo Shard already committed, `BossHatchService.Roll` already executed) sitting unresolved until the player supplies a name — the same "paid/won, not yet resolved" shape `PendingWeave` already solved, so it gets the identical treatment: `Sessions/PendingBossHatch.cs` (just the rolled `AnimaGenome`) + `PendingBossHatchEntity`/`PendingBossHatchRepository` (EF Core migration `AddPendingBossHatch`, applied and round-trip-verified against the real dev Postgres container, including surviving a simulated reconnect) + `PlayerSessionRegistry.CreateAsync` reloading it on every (re)connect + `GameHub.StartDelve` refusing to start a new Delve while one is unconfirmed (same guard shape `AttemptWeave` already has for a second pending Weave). Arguably even higher-stakes than a pending Weave: a Weave can be re-attempted for more Wisp, but a specific Boss clear cannot be re-fought for free.

**`SubmitAction`'s terminal-outcome handling (Phase 5a's stopping point) now resolves rewards inline, in the same call that reaches the terminal outcome** — same "no separate claim step, no double-claim window" pattern `CollectResourceNode`/`ClaimTreasureNode` already use, rather than a follow-up method the client might never call:
- **Victory (Combat/Elite):** `GrantNonBossVictoryRewardAsync` grants Wisp/Ember (queued through the existing pending-Ember flow)/Elite's 25% Vessel Shard, diffing the ledger before/after (`RewardService` grants straight into the ledger, doesn't return amounts) — surfaced on the new `CombatStatus.VictoryReward` field. The node clears, `ActiveCombat` clears, `ActiveDelveRun` stays alive (more map left to play).
- **Victory (Boss):** `GrantBossVictoryRewardAsync` grants Wisp/Echo Shard, bumps `RecordWinAsync` for every currently-held Artifact, rolls the hatched genome and stashes it as `PendingBossHatch` — surfaced on `VictoryReward` + the new `CombatStatus.BossHatchPreview` field (reuses `WeaveGenomePreview`, the exact same shape Weaving's Reveal screen already uses, via a new `ToPreview(AnimaGenome, hybridTriggered = false)` overload). `ActiveDelveRun` is cleared here too — Boss is the map's terminal node, nothing left to visit past it.
- **Defeat:** `DelveEndService.ResolveDefeat`'s 50%-keep math, surfaced on the new `CombatStatus.DefeatSummary` field (`WispEarnedThisRun`/`WispKept`/`WispForfeited` + `FloorIndexReached`/`NodesCleared`, the latter two captured BEFORE `MarkCurrentNodeCleared` so the summary reports nodes actually cleared before the fatal fight, not counting it). `ActiveDelveRun` is cleared.

**New `GameHub` methods:**
- `ConfirmBossHatch(name)` — mirrors `ConfirmWeave` exactly: materializes via `AnimaMaterializationService.Create` only now (not at Victory time), adds to `SanctumRoster`, persists, clears `PendingBossHatch`. No "Twin" concept (Boss-hatch always produces exactly one Anima).
- `GetPendingBossHatch()` — reconnect support, mirrors `GetPendingWeave`.
- `RetreatFromDelve()` — voluntary mid-run exit. **Guard: rejects with a `HubException` if `Session.ActiveCombat is not null`** (per the locked design: valid only "standing on the map between nodes," never mid-combat/inside a node) — no other "mid-node" state exists to guard against, since every other node type (Resource/Treasure/Shop) resolves atomically in one call and leaves nothing lingering. Calls `DelveEndService.ResolveRetreat` (0% penalty, no ledger mutation at all since nothing is forfeited), clears `ActiveDelveRun`. Returns the same `DelveEndSummary` shape Defeat uses. No separate `GetDelveEndSummary` method was added — both Defeat and Retreat return their summary inline, matching every other reward flow's "no follow-up getter" convention; a dropped connection right as either resolves loses at most the pretty summary screen, not the underlying economic effect (already committed/written through before the response is built).

**Deliberately NOT built this pass (flagged, not scope-crept into):** the Boss Victory "Delve Complete" appended summary (floors reached/Anima used/total Wisp earned, per the Match Result & Retreat System's own design) was not wired up — the task's own explicit scope for Boss Victory was Wisp/Echo Shard/Boss-hatch/win-count, not this. Worth noting it would be nearly free to add later: `DelveRun.Team` already IS "Anima used" and `DelveRun.WispEarnedSoFar`/`CurrentNode.FloorIndex` already exist, so no new tracking is needed, just a DTO + capturing them before `ActiveDelveRun` is nulled. Anima Profile's "Delve History" (a persisted per-Anima log of past Delves) is a SEPARATE, larger gap from the above — genuinely not a small addition (would need a new entity/migration/repository, plus a decision on what counts as a loggable event across Victory/Defeat/Retreat) — left as an explicit, still-open gap rather than built partially.

- **Boss Victory → Anima Reveal → ConfirmBossHatch**, using a directly-forced enemy-HP-to-0 shortcut to reach Victory (this harness verifies reward resolution, not combat balance — per CLAUDE.md's own note, the starter trio genuinely cannot beat Warden of the Hollow through legitimate play). `src/Anima.Server` is a real ASP.NET Core project, not a stub. Built this session:
- **Auth** — `AuthController`/`AuthService`: register, login, password-reset request/confirm. Deliberately minimal (not full ASP.NET Identity) — `PasswordHasher<T>` reused only for PBKDF2 hashing. `JwtTokenService` issues bearer tokens (180-day lifetime).
- **DB persistence** — EF Core, now on **Postgres** (Npgsql provider, was SQLite earlier this session and has since been switched — see docker-compose below). `AnimaDbContext` with `AccountEntity`/`PersistedAnimaEntity`/`PersistedLedgerEntryEntity`/`PasswordResetTokenEntity`. `SanctumRosterRepository`/`PersistentLedgerRepository` load/save per-account, `AccountLockRegistry` serializes writes per account.
- **SignalR** — `GameHub`, JWT-authenticated, including the `access_token` query-string workaround for transports (browser/Godot Web) that can't set a custom header on the handshake. Per-connection `PlayerSession` loads the account's DB-backed roster/ledger/active-team selection/pending Weave on connect, plus holds an in-memory free-Ember queue and the current Shop node's rolled stock for the current connection (see Ember & Augment System and Room-encounter screens above). Pending PURCHASED Embers are DB-backed but deliberately NOT session-cached — `GameHub` just queries `PendingPurchasedEmberRepository` directly each time, since every touchpoint was already async.
- **Local Postgres** — `deployment/docker-compose.yaml` runs a `postgres:16` container (`anima-postgres`, port 5432, db `anima_dev`, user `anima`), credentials sourced from a gitignored `deployment/.env` (see `deployment/.env.example`). `appsettings.json`/`appsettings.Development.json` no longer contain the real connection string or JWT signing key (checked-in secrets, fixed in a follow-up pass) — local `dotnet run` picks both up from `dotnet user-secrets` instead (one-time setup documented in `src/Anima.Server/README-dev-secrets.md`), and a real deployment would supply them via `ConnectionStrings__Default`/`Jwt__SigningKey` env vars. `Anima.Server` itself is not yet containerized (no Dockerfile/service block in `docker-compose.yaml`).

**`GameHub` method surface, current state (Sanctum/Collection, then Weaving, then Resource/Treasure, then Shop, then Combat core loop, then Combat rewards/Boss-hatch/Match Result — six separate sessions):**
- Foundation: `GetRoster`, `GetLedger`, `StartDelve` (refuses to start while a `PendingBossHatch` is unconfirmed, Phase 5b), `GetDelveStatus`.
- Sanctum/Collection: `GetRoster` also returns each Anima's parts list and `InTeam` flag; `SetTeam` persists the active-team selection to `AccountEntity.TeamAnimaIdsJson`; `GetArtifactCollection` returns the full 12-Artifact catalog joined against per-account discovered/won-with-count stats (`Discovered` is real as of Phase 3/4, see Artifact Cap above; `DelvesWonWith` still always 0).
- Anima Profile: `GetAnimaDetail` returns R1/R2 per part (the "Show hidden" toggle) plus resolved Parent/Echo-Twin names.
- Weaving: `AttemptWeave` (runs the real roll, commits Wisp/Echo Shard cost + WeaveCount immediately, persists the pending result so it survives a reconnect — see Weaving System above) and `ConfirmWeave` (the mandatory naming step — only now does the new Vessel's roster row get persisted); `GetPendingWeave` re-renders the Reveal screen after a reconnect.
- Delve traversal: `MoveToNode` (a real gap Phase 3 found — `DelveRun.TryMoveTo` existed with nothing calling it).
- Resource/Treasure: `CollectResourceNode`/`ClaimTreasureNode` (both mark the node cleared INSIDE the same call that grants the reward, so a client can't double-claim by calling again — see their own comments).
- Ember pickup-choice flow (shared by Resource/Treasure/Shop drops): `ConvertPendingEmberToWisp`/`AugmentPendingEmber` resolve one combined queue's front -- paid (Wares-bought, DB-backed) Embers first, then free (node-dropped, in-memory) ones, see Ember & Augment System above.
- Shop: `GetShopStock`/`RestAtShop`/`BuyWaresEmber`/`BuyWaresArtifact`, all routed through a shared `EnsureShopVisited` so the Wares roll and `ArtifactService.OnNodeVisited` fire exactly once per visit regardless of which action comes first. No `LeaveShop` method — nothing server-side needs to happen on Leave, the client just stops interacting.
- Combat (Phase 5a core loop + Phase 5b rewards, see both sections above): `StartCombat`/`SubmitAction`/`GetLegalTargets`/`GetCombatState`. `SubmitAction` now resolves Victory (Combat/Elite/Boss)/Defeat rewards inline the moment a winner is determined, via `CombatStatus.VictoryReward`/`BossHatchPreview`/`DefeatSummary`.
- Boss-hatch (Phase 5b): `ConfirmBossHatch`/`GetPendingBossHatch` -- the mandatory naming step for a Boss Victory's guaranteed hatched Anima, mirrors `ConfirmWeave`/`GetPendingWeave`.
- Delve end (Phase 5b): `RetreatFromDelve` -- voluntary mid-run exit, rejected while `ActiveCombat` is set.
- **Reforge is still not ported to hub methods** — deliberately deferred (0% map odds, unresolved open items: the Crest-slot bug, the Wisp-tier question, screen visual identity).
