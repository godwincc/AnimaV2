## What This Project Is

A breeding roguelike deckbuilder ("Anima") inspired by Axie Infinity (breeding/parts) and Slay the Spire (roguelike run structure). Players breed creatures called **Anima** — framed as constructs/"Vessels," not literal animals — to obtain skill combinations, build a 3-Anima team, and run a node-based roguelike dungeon.

**Core design pillar:** reward SMART breeding, smart team composition, smart deckbuilding — not just raw stats.

## Reforge (REDESIGNED, then REORDERED + PORTED TO GAMEHUB this session, alongside its map odds going from 0% back to 5% — see Map Generation above)

**Replaces the old RollOffer-then-Accept random-reveal flow entirely — there is no random part roll anywhere in the flow.** The browse order was FLIPPED this session from Part-first to **color-first**:
1. Player enters Reforge (or Leaves) and picks a color — Crimson/Onyx/Verdant/Azure. **The in-game UI calls this step "Aspect."**
2. Player **browses every Head/Frame/Tail skill for that color** (9 options: 3 Archetypes for that color × 3 Parts — `ReforgeService.GetBrowseOptionsByColor`), unfiltered by any target's currently-equipped skill (the target isn't picked until the next step, so that exclusion can no longer run here). **Crest is still excluded, not deferred** — it's the only part with no deck cards (`Models.Anima.DeckSkills` only ever draws Head/Frame/Tail), so it's structurally never in `ReforgePartPool.All` to begin with, for any color.
3. Player picks which of their (up to) 3 team Anima to apply it to. Part is now **fully determined by which skill was picked** — there's no separate "pick a slot" step anymore. The exclusion that used to happen at browse time moved here instead: if the picked skill would be a no-op for a given Anima (already equipped in that Part, whether from its own real genome or an already-Accepted override earlier this Delve), that Anima is skipped/disabled as a target rather than allowing a pointless swap (`ReforgeService.IsNoOpForTarget`/`GetValidTargets`).
4. A confirm-only step shows the slot being overwritten (Head/Frame/Tail, read off the picked skill) and the exact Wisp cost for the picked target, then Accept/Decline.
5. **Accept commits a run-scoped override on `DelveRun`** (`DelveRun.SetReforgeOverride`/`GetEffectiveSkill`/`GetEffectiveDeckSkills`), keyed by (target Anima's Id, Part) — it does **NOT** mutate the target `Models.Anima`'s real `Head`/`Frame`/`Tail` fields, for the same reason as before (that instance is the shared HP-attrition/DB-persisted reference). The override is consulted only at deck-build time via `CombatEngine`'s optional `DelveRun?` constructor param, unchanged by this session's reorder. R1/R2 and all Weaving/lineage data are never read or written. Decline is free, nothing changes (mark the node cleared, nothing else).
6. **Cost, re-verified this session, unchanged:** 40 Wisp if the picked skill's color matches the target Anima's own body color, 80 Wisp otherwise; hybrids (Vulcan/Mirage) ALWAYS cost 80 regardless of pick. Confirmed by reading the code, not assumed: `GetAcceptCost` only ever needs both values (skill color, target color) at Accept time, so the order they were picked in during the flow doesn't matter — still correct now that color is chosen before the Anima.

**Insufficient-Wisp path (NEW this session): a distinct rejection, not a generic exception.** `ReforgeService.Accept` now returns a `ReforgeAcceptResult` (`Outcome`: `Success`/`InsufficientWisp`, plus `Cost`/`WispBalance`) instead of a bare bool — deliberately different from the Shop hub methods' existing "throw `HubException(\"Insufficient Wisp.\")`" convention, asked for explicitly here so the client can show "not enough Wisp — needed X, have Y" and route back to the Reforge/Leave landing screen. Nothing partially commits on this path: no Wisp is deducted, no override is recorded, and (at the `GameHub.AcceptReforge` layer) the node stays uncleared, so the player can retry with a cheaper pick or Decline at the same node visit with no leaked state.

**Terminology retirement (flagged explicitly, not a silent rename):** "Aspect" now means the picked COLOR in the UI. The OLD design's first step was "pick an Aspect: Head, Frame, or Tail" — that meaning is fully retired, not kept alongside the new one. Every comment/variable name in `Anima.Core.Reforge.ReforgeService` that used to mean the old thing (Part) has been updated to say "Part" explicitly, so nothing in the codebase still says "Aspect" meaning Head/Frame/Tail now that the UI says "Aspect" meaning color.

**Discrepancy found and resolved during the original redesign (unaffected by this session's reorder):** the deck only ever draws 3 copies of a part's single Dominant skill (`CopiesPerSkill = 3`) — R1/R2 are pure hidden-genome data, never deck cards, so the override replaces just the effective Dominant used at deck-build time.

**Regression coverage (from the original redesign, re-confirmed this session under the new API): Reforge override survives across multiple combats within the same Delve.** The deck rebuilds fresh every `StartCombat()` call, so the override has to be re-read from `DelveRun`'s dictionary every time. Harness check: Reforge once via `GetBrowseOptionsByColor`/`Accept`, then win two entirely separate combats (fresh `CombatState`/`CombatEngine` each time, same `DelveRun`/team both times) — confirms the override's skill is present in both fights' freshly-built decks.

**Screen visual identity: RESOLVED this session, no longer deferred.** No distinct Reforge palette after all — reuses the same standard warm sanctuary/workshop gradient every other core screen uses (the previously-proposed white/pale theme is dropped). A hammer icon (matching the existing map-icon convention where hammer=Reforge) is the screen's visual accent: small in the header throughout the flow, larger/centered on the initial Reforge/Leave landing screen where there'd otherwise be empty space.

**Ported to `GameHub` this session, for the first time** (alongside the map odds going from 0% to 5% — a 0%-odds mechanic had nothing real to reach it, so hub wiring had no urgency until now): `GetReforgeBrowseOptions` (color → 9 skill options)/`GetReforgeValidTargets` (skill → per-target Wisp cost, no-op targets excluded)/`AcceptReforge` (commits or returns `InsufficientWisp`)/`DeclineReforge` (marks the node cleared, nothing else). `EnsureReforgeVisited` fires `ArtifactService.OnNodeVisited` exactly once per node visit (mirroring Shop's `EnsureShopVisited`), but — unlike Shop — is deliberately NOT coupled to `MarkCurrentNodeCleared`: browsing must never clear the node (the player can look at multiple colors/skills/targets before committing), and per the insufficient-Wisp path's own requirement, a rejected `AcceptReforge` must leave the node resolvable again. Only a successful `AcceptReforge` or an explicit `DeclineReforge` ever clears the node.

## Weaving System (LOCKED, VERIFIED — real Axie Infinity mechanics as direct inspiration)
Per part slot (ALL FOUR: Head/Frame/Tail/Crest, no exceptions): pool 6 candidate genes (D/R1/R2 × 2 parents), weighted 37.5%/9.375%/3.125% → winner becomes offspring's new Dominant. Two more rolls fill R1/R2. Mutation: 10%, R1/R2 only. Color: flat 50/50, 100% if same-color parents. Hybrid trigger: both parents pure + correct pairing = 33%.
Sibling restriction (copies Axie): only parent-offspring + full-sibling blocked. Weave Count: max 5, cost curve 50/100/175/275/400. Echo: twin-Vessel, 5% spontaneous or guaranteed via 5 Echo Shards.
**Naming:** players name their own Animas — mandatory prompt on Weave creation. **The old hardcoded-starter-trio exception (Ember/Boulder/Sprout, no prompt) is RETIRED this session** — every new account now gets a real rolled trio through the same mandatory-naming flow as everything else. See Starter Trio (Real Roll + Naming Flow) below.

**R1/R2 discard BUGFIX (found + fixed a previous session, predates this session — not newly introduced):** `AnimaMaterializationService.Build` (both the Weave and Boss-hatch overloads) only ever carried each part's Dominant skill into the materialized `Models.Anima`, silently dropping R1/R2 the instant a Vessel was created. This wasn't just a Profile-display gap — a materialized Anima that later became a parent had no R1/R2 of its own for `WeavingService.ResolvePart`'s 6-gene weighted pool to draw from, so **every 2nd-generation Weave was silently working off a broken (Dominant-only) pool** until this fix. Fixed by: (1) adding nullable `HeadR1`/`HeadR2`/`FrameR1`/`FrameR2`/`TailR1`/`TailR2`/`CrestR1`/`CrestR2` fields to `Models.Anima` (nullable, not `required` — see the Starter Trio section below for what stays nullable now that the real starter trio also goes through `Build`); (2) `Build` now populates all three genes per part instead of just Dominant; (3) `GenomeFactory.CreateGenome(anima)` is the single entry point for "get this roster Anima's `AnimaGenome` so it can be a Weave parent" — real stored R1/R2 (`ExtractGenome`) for anything materialized via `Build` (now including the real starter trio, see below), falling back to the pre-existing `CreateFounderGenome` synthesized placeholder only for bare `SampleAnimas.CreateX()` test fixtures (detected via `HeadR1 is null`). No EF Core migration was needed for any of this: `PersistedAnimaEntity` stores the whole `Anima` as an opaque `AnimaJson` blob (same as the pre-existing `WeaveCount`/`ParentAId` fields), so the new nullable fields flow through automatically — verified for real with a live round-trip against the dev Postgres container, not just a design-time check.

**Echo Twin link (NEW this session):** `Models.Anima.EchoTwinId` (nullable string) is now set on both sides of an Echo-triggered Weave's Primary/Twin pair (by `GameHub.ConfirmWeave`, after both are materialized and their real Ids exist). This closes a real gap: `WeavingService.AttemptWeave`'s Twin is bred from the exact same parent pair as its Primary in the same call, so before this fix a genuine Echo Twin was indistinguishable from an ordinary full sibling via `ParentAId`/`ParentBId` alone (both would just look like two full siblings — correctly so, since an Echo Twin genuinely IS a full sibling too, just additionally linked). `WeavingService.AreFullSiblings`/`IsDirectParentChild` are untouched and still validate correctly against the new fields (they only ever read `ParentAId`/`ParentBId`, which `EchoTwinId`/R1/R2 don't affect) — confirmed via the existing 140-check console harness suite, still 140/140 after this change.

**Weaving is now ported onto `GameHub`** (see Server / Accounts / Auth below) — `AttemptWeave`/`ConfirmWeave`, both real economic effects (Wisp/Echo Shard spend, WeaveCount increment) committing immediately in `AttemptWeave`, with only the resulting Vessel(s)' roster row waiting on `ConfirmWeave`'s mandatory naming step (matching the Anima Reveal screen's "name is not skippable" contract).

**Pending-Weave persistence gap (found + fixed, Phase 3 audit):** the guard that blocks a second `AttemptWeave` while one is unnamed used to live only on `PlayerSession` (in-memory, keyed off a SignalR `ConnectionId`) — a dropped connection between `AttemptWeave` and `ConfirmWeave` dropped the whole `PlayerSession` too, silently resetting the guard AND permanently losing the already-paid-for roll (Wisp/Echo Shard spent, a WeaveCount charge consumed, no Vessel to show for it — WeaveCount is capped at 5 uses per parent, so this was a real, non-trivial loss). Fixed with a new `PendingWeaveEntity`/`PendingWeaveRepository` (EF Core migration `AddPendingWeave`, applied and round-trip-verified against the local Postgres dev container, including a real `WeavingResult` genome) — `AttemptWeave` now persists the pending Weave immediately, `PlayerSessionRegistry.CreateAsync` reloads it on every (re)connect, and a new `GetPendingWeave` method lets a reconnected client re-render the Anima Reveal screen before confirming. `ConfirmWeave` deletes the row on success. Deliberately NOT extended to `DelveRun` or pending Ember pickups in the same pass — see their own sections for why those stayed in-memory-only (lower stakes, not the thing that was actually broken).

## Silent Chime (REDESIGNED THIS FOLLOW-UP SESSION — replaces the old "grants an extra action" effect entirely)
**New effect — Enrage Interrupt, fully automatic, no player action/targeting of any kind:** single-use per Delve. The first time any enemy (Elite or Boss, whichever comes first) triggers Enrage activation in a Delve where the player holds Silent Chime, the Artifact automatically cancels that Enrage activation entirely (it never applies — the enemy simply stays un-enraged for that attempt) and is consumed. Any Enrage activation after that — the same enemy re-attempting the very next Round (see below), or a different enemy later in the same Delve — resolves normally, since the charge is already spent.

**Confirmed before implementing (as asked):** Enrage activation is a single per-enemy event, gated by `!enemy.IsEnraged` inside `CombatEngine.RoundStartPhase` — once `IsEnraged` is set true, `GetEnrageMultiplier` only ever passively reads it afterward (the ever-doubling bonus is pure math off `EnrageTriggeredRound`, no further "trigger" check). However, the ACTIVATION check itself (the `RoundStartPhase` block that flips `IsEnraged`) genuinely does re-run every Round for every not-yet-enraged eligible enemy — so if Silent Chime cancels an attempt by deliberately leaving `IsEnraged` false, that exact same enemy's condition is re-evaluated (and can succeed) on the very next `RoundStartPhase` call. This is real, intentional re-triggering, not a bug — confirmed to match the locked design's own "whether from the same enemy re-triggering... or a different enemy later" framing.

**Implementation:** `CombatEngine.TryConsumeSilentChimeCancel()` (new, private) is called from inside `RoundStartPhase`'s existing Enrage-check loop, immediately before `enemy.IsEnraged` would be set — if it returns true, the loop `continue`s for that enemy instead (no `IsEnraged`/`EnrageTriggeredRound` write at all). Gated on `DelveRun.CurrentArtifacts` (held) and consumption removes the Artifact from the real `DelveRun.RunLedger.Artifacts` list (same "consumed = removed" pattern Withering Fang already uses in `ArtifactService.OnNodeVisited` — not a separate flag, so "held" and "not yet consumed" are the same check by construction). This requires a real `DelveRun` (not just `CombatEngine`'s own defensively-copied `_ownedArtifacts` snapshot) because the charge must survive into whichever LATER combat in the same Delve is the one that actually triggers Enrage first — a fresh `CombatEngine` is reconstructed per combat (per hub call, in `GameHub`'s real pattern), so anything not stored on `DelveRun` itself would silently reset between fights. Exactly the same cross-combat persistence problem Reforge's own `DelveRun`-scoped override already solves (and covered by the same style of regression test — see below). If no `DelveRun` was supplied (an old call site), this silently no-ops.

**No `GameHub` method is needed for this at all, confirmed and acted on.** The old, never-wired `TryActivateSilentChime`/`PendingSilentChimeTarget`/`AdvanceTurnOrConsumeSilentChime` activation surface is deleted entirely, not merely left unwired — audited first to confirm nothing else depended on them (only `CombatEngine`/`CombatState`/the console harness referenced any of the three; no `GameHub` method ever did). This resolves the long-standing "Silent Chime not wired to any `GameHub` method" known gap by removing the need for wiring, not by adding it — there is no longer any player-facing activation step to wire up. `ActionPhase`'s extra-action injection block and the resumable path's `AdvanceTurnOrConsumeSilentChime` helper are both gone; `ResolvePlayerAction`/`ResolvePlayerPass` now just do a plain `_state.TurnIndex++` again, identical to how every other Artifact-free turn already advanced.

**Verified via new console harness checks** (replacing the old 3 "grants an extra action" checks with 4 new ones, net +1, full suite still 100% green after the change): (1) the first Enrage activation is cancelled entirely and the Artifact is consumed immediately; (2) the SAME enemy re-attempts and enrages for real the very next Round once the charge is spent; (3) baseline sanity — without Silent Chime, Enrage triggers normally; (4) **cross-combat persistence** — Silent Chime survives a full Enrage-free Combat 1 unconsumed (fresh `CombatState`/`CombatEngine`, same `DelveRun`), then correctly cancels Enrage in a separate Combat 2 later in the same Delve.

### Turn Resolution (LOCKED THIS SESSION — Darkest Dungeon-style, not STS batch-commit)
Turn order for the Round is rolled once at Round start (all 6 combatants), shown as a queue. Combatants act **one at a time in that rolled order**, interleaving player and enemy turns — NOT a "queue all your actions, then watch the round resolve" STS-style batch. This was a deliberate choice: the skill kit has heavy reactive/interrupt design (Intercept, Guard Strike, Deflect, Ward, Retribution, Vengeance, Last Laugh) that only makes sense if the player can see what already happened this Round before acting. Enemy intent telegraphing (STS-style "show what the enemy will do before they do it") was explicitly REMOVED given one-by-one resolution — the player sees the real action play out live instead.

Player can **Pass** a given Anima's turn (opts out of playing anything that turn) via an explicit Pass action.

## DelveRun — Run/Traversal State Model (NEW THIS SESSION — implemented)

`Anima.Core.Run.DelveRun` is the missing glue between node visits — the "save file for a single in-progress run." Scope was deliberately kept minimal (in-memory only, no save/resume, no mid-combat state):

- **`CurrentNode`/`AvailableNodes`/`TryMoveTo`** — map position on the 7×15+Boss grid. Boss surfaces automatically as an available node with no special-casing needed.
- **`ClearedNodes`** — caller-driven (not automatic on move), so the map can render visited vs. available.
- **`Team`** — a plain `List<Anima>`, the same list a Combat node's `CombatState.PlayerTeam` is built from. HP attrition needed zero separate tracking as a result — `CurrentHp` mutations persist for free through the shared reference, **as long as nothing ever rebuilds/clones this list** (see gotcha above).
- **`CurrentArtifacts`/`WispEarnedSoFar`** — live passthroughs to `RunLedger.Artifacts`/`PersistentLedger`, not separate copies, to avoid forking state that has to be hand-synced.
- **`DelveRun.Start(...)`** — the one real call site that sets `RunLedger.WispAtDelveStart` from the current `PersistentLedger` balance. **Bugfix note:** `WispAtDelveStart` existed since the Match Result & Retreat session but nothing outside test code ever actually set it — every real Delve would have diffed Wisp-earned against a stale 0 before this fix.

**Explicitly out of scope for this pass (by design, not oversight):** no mid-combat state (an active Combat node owns its own state; `DelveRun` just receives the result on resolution), no save/resume across app restarts (matches `SanctumRoster`/`PersistentLedger`'s existing no-save/load caveat), no multiplayer/concurrency handling (single active `DelveRun` at a time).

## Combat System — Server Port (Phase 5a, NEW THIS SESSION — core loop only, no rewards)

Ports the core Combat loop onto `GameHub`. **Deliberately stops the moment a winner is determined** — Victory/Defeat rewards (Wisp/Ember/Vessel/Echo Shard), `DelveEndService`, Boss-hatch + Anima Reveal, and the Artifact win-count stat are all Phase 5b, not touched here.

**Audit findings (confirmed by reading `CombatEngine`/`CombatState` and the console harness's own `RunDelveFight`, not assumed):**
1. **`RunRound()` is one synchronous, uninterruptible loop** — `RoundStartPhase → InitiativePhase → ActionPhase (loops every combatant, both sides, interleaved) → RoundEndPhase`, all in a single call. Player decisions came from a synchronous `Func<Anima, CombatState, Skill?>` callback (`ChoosePlayerSkill`) invoked mid-loop — fine for the console harness's scripted heuristic AI (`ChooseDelveSkill`), architecturally incompatible with a real human player needing an actual network round-trip per decision. This is genuine restructuring work, not a thin wrapper — confirmed, not assumed, before writing any hub code.
2. **No real player target choice existed anywhere in Core.** `CombatEngine.SelectTarget` is a fully deterministic static method (Marked-priority → front position, or lowest-HP, or first-alive). `TargetType.ChosenEnemy`/`ChosenAny` exist as enum values, and `SampleAnimas.cs`'s own comments literally describe them as "stands in for a real player choice" — but `SelectTarget`'s switch never distinguished them from the fully-automatic types; they fell through to the same `_ => alive.FirstOrDefault()` default. `ChoosePlayerSkill`'s signature only ever returns a `Skill`, never a target — the console harness's `ChooseDelveSkill` never picks one either. A second, independent architectural gap from turn-pausing.
3.
